# Source syntax: the locator IS the source string; tag resolution is host-agnostic

`[dependencies]` entries take a single `source` value that encodes the locator directly — `owner/repo` (default GitHub), `gitlab:owner/repo`, `bitbucket:owner/repo`, `https://host/x.tar.gz` (arbitrary tarball), or a path (`./foo`, `../bar`, `/abs/foo`, `~/foo`, or `local:./foo`). There is no `source = "github|gitlab|bitbucket"` selector and no separate `repo` / `path` key. The bare-string shorthand `name = "<source>@<spec>"` returns; the inline-table form (`{ source = "...", version = "...", include = [...], exclude = [...] }`) goes through the same parser. The earlier single-string `subdir = "..."` field is removed — see "include / exclude on dependencies" below. Version pinning uses one field, `version`, which accepts a SemVer 2.0.0 range (`^1.0.0`, `>=1.0.0 <2.0.0`), an exact SemVer 2.0.0 version (`1.0.0`), a commit SHA (7–40 hex chars), or an arbitrary Git tag name (`v1.0.0`, `release-2024`, `nightly`). Tag resolution for SemVer-shaped specs goes through git smart-HTTP's `info/refs?service=git-upload-pack` endpoint — works against every git host with one URL pattern, returns refs in a small pkt-line text format, no JSON parsing, no auth tokens needed for public repos. This replaces the earlier design where every dep declared `source = "github|gitlab|bitbucket|release|local"` separately from its `repo`/`path` value and tag resolution didn't exist (the user pinned to an exact `ref = "..."` per dep). We chose this over keeping the explicit-kind selector because the locator carries the kind unambiguously, over per-host REST APIs because smart-HTTP is uniform across all git hosts (including self-hosted Gitea/Gogs/Forgejo/Bitbucket Server), and over `git clone` because LWPT's "HTTP archive endpoints, not git protocol" constraint (handoff scope choice) is preserved — `info/refs` is HTTP-only, no git binary required.

## Considered Options

- **Keep the earlier explicit `source = "..."` field.** Familiar to anyone who has read the spike's manifests, no migration. Rejected because it's redundant — the `repo` / `path` value already tells you the kind in every realistic case, and the lockfile loses host info anyway (the earlier `sourceType = "github"` collapses gitlab + bitbucket into "github"). The two fields disagreeing was a real footgun.
- **Use per-host REST APIs for tag listing (GitHub `/repos/<slug>/tags`, GitLab `/api/v4/projects/<urlencoded-slug>/repository/tags`, Bitbucket `/api/2.0/repositories/<slug>/refs/tags`).** Returns JSON which is well-specified per host. Rejected because (1) we'd need a JSON parser (LWPT has only the partial TOML reader), (2) each host has different field names + pagination + auth-for-rate-limit, (3) self-hosted git instances would each need bespoke support. The smart-HTTP path collapses all of this into one URL pattern + one tiny parser.
- **`git clone` for tag listing AND for fetching.** The most "git-native" approach. Rejected for the handoff's reason: LWPT's value prop is single-binary RTL-only deployment; adding `git` as a runtime dep breaks that. The HTTP archive endpoint approach (preserved here) gives LWPT direct control over the tarball bytes for hash verification.
- **Treat `v1.0.0` as a SemVer 2.0.0 version that we strip the `v` from.** Convenient — every common Git tagging convention "just works". Rejected because [SemVer 2.0.0 explicitly says](https://semver.org/#is-v123-a-semantic-version) `v1.2.3` is not a SemVer — it's a Git tag name that happens to contain one. Interpreting a `v`-prefixed spec as SemVer imports the Git tagging convention into the semantic layer where it doesn't belong. The user-side asymmetry is intentional: `version = "1.0.0"` is "I want SemVer 1.0.0; resolve through whichever tag the repo uses (with or without `v`)" and `version = "v1.0.0"` is "I want the literal Git tag `v1.0.0`; no SemVer magic."
- **Add a `host` field as a sibling of `source` in the lockfile.** Would unambiguously record gitlab + bitbucket without inferring from the locator string. Rejected because the locator string is the source of truth — the host is recoverable by re-parsing it (which `LoadLockfile` does, via `ParseDependencySource`). Adding `host` as a separate field invites it to disagree with `source`.

## Consequences

- **Manifest format breaks at v0.x.** No migration code — the manifest reader hard-errors on the earlier shape (explicit `source = "github|gitlab|bitbucket|release"` literal OR any of `repo` / `ref` / `tag` / `asset` / `path` keys present) with a message naming the new form. Pre-release: acceptable. The error message references this ADR.
- **`release` source kind is gone.** Release-asset downloads (precompiled binaries posted as GitHub release assets) now go through the URL form: `source = "https://github.com/<owner>/<repo>/releases/download/<tag>/<asset>"`. The semver-against-Git-tags path handles the common case (source archive at a tagged release) without a dedicated kind.
- **Lockfile schema bumps to v3.** Old `sourceType` field is gone (host inferable from the source string + the resolved URL); `source` field holds the manifest's verbatim source string; new `resolvedRef` records the concrete tag/SHA the resolver picked; new `resolvedURL` records the actual archive URL fetched. Earlier lockfile schemas migrate via one-shot `rm + re-install`; we're pre-release so this is fine, and v3 is the last schema break planned for v1.
- **`v`-prefixed specs are arbitrary Git tags.** Writing `version = "v1.0.0"` opts out of SemVer-style resolution: we look for the literal tag `v1.0.0`. Writing `version = "1.0.0"` opts in: we look for tag `1.0.0` first, then `v1.0.0` as fallback. Both shapes commonly resolve to the same tag in practice; the difference is the manifest-author's intent (SemVer-aware vs literal).
- **Tag listing happens before every install of a SemVer-shaped dep.** One extra HTTP round trip per dep for the `info/refs` GET. No caching in v1 — the response is small (typical repo: < 50 KB) and the resolver only runs on `lwpt install`, not on `lwpt build` / `format` / `test`. A v1.x caching pass would key on (repo URL, last-resolved-time) and respect a `--no-cache` flag.
- **Self-hosted git works via the `[sources]` table.** Users declare custom hosts in a single `[sources]` section, one inline-table per prefix:

  ```toml
  [sources]
  gitea   = { archive = "https://git.example.com/{user}/{repository}/archive/{ref}.tar.gz", git = "https://git.example.com/{user}/{repository}.git" }
  forgejo = { archive = "https://forgejo.example.com/{user}/{repository}/archive/{ref}.tar.gz", git = "https://forgejo.example.com/{user}/{repository}.git" }
  ```

  Then any dep can use the prefix: `mylib = "gitea:team/mylib@^1.0.0"`. Three placeholder strings are supported: `{user}`, `{repository}`, `{ref}` (the resolved tag / SHA / branch). The actual placeholder text lives in the `PLACEHOLDER_USER` / `PLACEHOLDER_REPOSITORY` / `PLACEHOLDER_REF` constants in `LWPT.Core` so the syntax is changeable in one spot. The `archive` URL template assembles the .tar.gz download URL; the `git` URL template points at the smart-HTTP base (LWPT appends `/info/refs?service=git-upload-pack` for tag listing). Both fields are required at manifest-load time; templates that omit any required placeholder hard-error so the failure surfaces before a fetch attempt. The `[sources]` table is per-project (lives in the manifest, not user config); names that shadow built-ins (`github` / `gitlab` / `bitbucket` / `local`) are rejected at parse time.

  Lockfile entries for custom-prefix deps round-trip via the verbatim `source` field (e.g. `source = "gitea:team/mylib"`); LoadLockfile is permissive about unknown prefixes (it doesn't have `[sources]` context, so it accepts the prefix as `hkCustom` for diagnostics — verification is hash-based and doesn't need the URL templates back).

- **include / exclude on dependencies mirror the formatter's `[format]` scope.** A dep's inline-table form accepts `include = ["..."]` and `exclude = ["..."]` arrays of globs. Semantics match ADR-0007: include is additive on top of "the whole extracted archive"; exclude is subtractive on the resulting set; neither array set → keep every file (the common case). Globs are post-extraction file paths relative to the dep's modules dir. The same matcher (`MatchPathGlob`) supports `*` / `?` / `**` (recursive segment). The earlier single `subdir = "src/middleware"` form is replaced by `include = ["src/middleware/**"]`; the parser rejects `subdir` with a migration hint pointing at this ADR. Empty directories are reaped after pruning so the modules dir doesn't accumulate dead branches.

The decision **forecloses** the path back to per-host REST APIs and the explicit `source` selector. The decision **opens** smart-HTTP-based capabilities for the future: ref listing for `--no-network` consistency checks; tag-listing-as-a-subcommand (`lwpt info <dep>`); branch-pinning (already supported via the literal-tag path, since smart-HTTP returns `refs/heads/` too); and `[sources]` extensions like template variants for future hosts that don't fit the three built-in patterns.

### Footnote: SemVer 1.0.0 → 2.0.0 on the `v` prefix

SemVer 1.0.0 had a notion of "preferred" (unprefixed) vs "not preferred" (`v`-prefixed) forms of the version string. SemVer 2.0.0 dropped that hierarchy; the [2.0.0 FAQ](https://semver.org/#is-v123-a-semantic-version) is explicit that `v1.2.3` is not a SemVer at all. This ADR's rule (`v`-prefixed = literal Git tag, unprefixed = SemVer) is the SemVer-2.0.0-faithful interpretation; codebases that recognise both `1.0.0` and `v1.0.0` as "semver-y" are working from the older 1.0.0 spec.
