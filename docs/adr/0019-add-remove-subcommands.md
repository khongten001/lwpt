# `lwpt add` and `lwpt remove` are the eighth and ninth subcommands

The subcommand surface, frozen at seven since [ADR-0013](./0013-run-subcommand-and-build-rename.md) ([Hard Constraint in AGENTS.md](../../AGENTS.md)), expands to nine with `lwpt add` and `lwpt remove`. Both are manifest-editing frontends to the existing install transaction ([ADR-0018](./0018-install-transaction-module.md)): `lwpt add <source[@version]> [--name <name>]` validates the bare-string source spec ([ADR-0009](./0009-source-syntax-and-tag-resolution.md) shorthand, the same string that lands in the manifest), stages a comment-preserving textual edit of `lwpt.toml`, and hands both to a *manifest-mutation transaction* — the install transaction extended so that, after the lockfile + cfg commit, it also commits the edited manifest (atomically through `.lwpt/tmp/`) and prunes orphans, all inside the same cross-process install lock. `lwpt remove <name> [<name>...]` is the inverse: it deletes the `[dependencies]` entries and runs the same mutation transaction on the remaining graph, which prunes orphaned `.lwpt/modules/<name>/` trees and their cached archives by diffing the pre-transaction lockfile against the freshly resolved graph. We chose this over keeping the surface at seven and documenting "edit `lwpt.toml`, then run `lwpt install`" because the edit-then-install loop is the single most common dependency workflow, the npm/cargo/yarn precedent (`npm install <pkg>` / `cargo add` / `yarn add`) is universal, and — decisively — manual removal leaves committed garbage behind: the install transaction regenerates `lwpt.lock` + `lwpt.cfg` from the manifest but never deletes module trees, so a hand-removed dep silently leaves its extracted tree and archive in the committed zero-install state forever.

## Subcommand contracts

- **`lwpt add <source[@version]> [--name <name>]`** — the positional is exactly the bare-string shorthand from ADR-0009 (`owner/repo@^1.0`, `gitlab:org/repo@sha`, `./vendor/leaf`, `https://.../x.tar.gz`). The dependency name defaults to the last path segment of the locator (the `repo` half of `owner/repo`; the basename of a local path). URL and `workspace:` sources have no usable basename, so they require `--name`. The name must satisfy the package-name grammar (ASCII letters/digits/hyphen/underscore) and must not collide with a discovered workspace package (which would race for the same `.lwpt/modules/<name>/` slot).
- **Re-adding an existing name updates its spec in place** (cargo-add precedent) and reports `updated` instead of `added`. If the existing entry is an inline table (`{ source = ..., include = [...] }`), replacing it with a bare string would silently drop the include/exclude globs — that case is a hard error directing the user to edit the manifest by hand.
- **`lwpt remove <name> [<name>...]`** — each name must be a direct `[dependencies]` entry. Removing a workspace package is a hard error pointing at `[workspaces]`; an unknown name is a hard error. A removed dep that is still reachable transitively through another dep's manifest stays in the lockfile and modules tree (with a notice) — only its *direct* declaration goes away.
- **Uneditable forms are hard errors in both directions.** A dep declared in a form the textual editor cannot handle — a `[dependencies.<name>]` dotted table — is a hard error directing to a manual edit, for `remove` AND for `add`. The `add` guard matters most: inserting a bare `name = "..."` line alongside an existing dotted table would define the key twice and brick the manifest on its next parse (the TOML reader hard-errors on duplicate definitions).
- **Install-before-write ordering, inside the lock.** The mutation transaction resolves and materializes the amended graph first and commits `lwpt.toml` only on success — a failed resolve/fetch (typo'd repo, dead tag, offline) leaves the manifest byte-identical, with no "manifest says X, lockfile says Y" intermediate state to repair. Because the manifest commit and the orphan pruning happen *inside* the cross-process install lock (same `TInstallLock` scope as the lockfile + cfg writes), a concurrent `lwpt install` can neither interleave between "lockfile written" and "manifest written" nor race the prune deletions. The textual edit preserves every comment, the file's line-ending style (CRLF manifests stay CRLF), and the trailing comment on an updated line; only the one `name = "..."` line is inserted, replaced, or deleted (plus the `[dependencies]` header when the section doesn't exist yet). Value escaping uses the same `TomlEscape` as the lockfile writer, so the two writers cannot drift.
- **Orphan pruning is lockfile-diff-driven.** The mutation transaction snapshots the previous lockfile right after acquiring the lock, and after writing the new one diffs the two: any package name present before and absent now gets its `.lwpt/modules/<name>/` tree and its cached `.lwpt/archives/` tarball deleted; a package whose resolved ref changed (an `add` that bumps a version) — or that moved to a local path, which has no archive — gets its stale old archive deleted. This is the one sanctioned exception to "only `lwpt install` touches the committed `.lwpt/` state" — `add`/`remove` ARE the install pipeline (same transaction, same cross-process lock), extended with the cleanup the transaction itself never needed while the manifest was hand-edited. The diff also reaps transitive orphans (a removed direct dep's now-unreferenced children).
- **Install lifecycle hooks fire.** `preinstall`/`postinstall` ([ADR-0011](./0011-build-lifecycle-hooks.md)) run around the transaction exactly as in `lwpt install` — both subcommands materialize dependency state, and a hook keyed to "deps changed" must not be skippable by choosing `lwpt add` over a manual edit + `lwpt install`.
- **Network posture:** `add` and `remove` are install-class subcommands — they may hit the network exactly as `lwpt install` does (tag listing + archive fetch for unfetched git-host/URL deps). The "network operations are explicit" boundary in AGENTS.md gains these two alongside `install`.

## Considered Options

- **Keep the surface at seven; document edit + `lwpt install`.** Smallest change; no ADR needed. Rejected because removal is not actually achievable by hand without violating a Hard Constraint: the transaction never prunes `.lwpt/modules/`, and the constraint forbids deleting from it manually — so "remove a dep cleanly" had *no* sanctioned path at all. `add` alone wouldn't justify the expansion; `remove`'s pruning gap does, and shipping `remove` without its dual is gratuitously asymmetric.
- **A TOML serializer: parse, mutate, re-emit the whole manifest.** Structurally clean, handles every authoring form. Rejected because the TOML package is deliberately read-only, and a round-tripping writer that preserves comments, blank lines, and key order is a large new surface (style-preserving TOML emission is the hard part of every `cargo add` clone) — all to support editing one line. The textual editor handles the canonical bare-string form plus single-line inline tables, and hard-errors with a "edit by hand" pointer on anything fancier. The escape hatch is always available; the 95% case is one clean line.
- **Manifest-write-first, then install (npm's observable order).** Simpler to reason about ("the manifest is the source of truth, install follows"). Rejected because a failed install then strands a manifest declaring a dep that was never fetched — `--frozen` CI fails, the user has to know to revert the line. Install-first means failure leaves zero residue.
- **`lwpt install <pkg>` instead of a new `add` subcommand (the npm spelling).** No surface expansion. Rejected because `install`'s contract is "materialize the manifest" and overloading its positional to mean "also mutate the manifest" muddies both contracts (`--frozen` + positional combinations become a semantics matrix). cargo's split (`install` materializes, `add` edits) maps cleanly onto LWPT's existing transaction; `add`/`remove` are thin frontends and `install` stays untouched.
- **Generalize pruning into `lwpt install` itself.** Every install would self-clean orphans, and `remove` could be pure manifest surgery. Tempting, and possibly right eventually — but it changes the semantics of the most load-bearing subcommand (an install after a hand-edit would silently delete committed state), which deserves its own deliberation. Deferred; pruning stays scoped to the two subcommands whose explicit job is "this dep is going away / changing".
- **`lwpt remove` deletes modules immediately, without re-running the transaction.** Faster, fully offline. Rejected because only the resolver knows whether the removed dep is still reachable transitively, and the lockfile + cfg must be regenerated anyway — skipping the transaction would reinvent its graph walk badly.

## Consequences

- **The frozen-surface Hard Constraint gains its ADR-cited expansion**: the canonical set is now `init`, `install`, `add`, `remove`, `build`, `format`, `test`, `repair`, `run` (plus the pre-approved `health` / `duplication` from ADR-0006). AGENTS.md, README, and quick-start update; `add` and `remove` join `RESERVED_SUBCOMMAND_NAMES` so a `[add]`/`[remove]` run-script section is a manifest-load error (ADR-0013 rule).
- **`lwpt.toml` joins the atomic-write set, owned by the transaction seam.** The manifest was previously only ever written by `init` (file creation, not mutation). `add`/`remove` mutate it via `RunManifestMutationTransaction` in `LWPT.Install`, which commits the edited lines through `AtomicWriteText` inside the install lock — extending ADR-0018's "lockfile/cfg commits are owned by the transaction seam" to the manifest for mutation flows. Hooks stay outside the seam (ADR-0018 unchanged).
- **The textual editor is a new shared unit (`LWPT.ManifestEdit`)** with co-located unit tests: section location, key matching (bare or quoted), insert/replace/delete, section creation, comment + line-ending + trailing-comment preservation, shared-`TomlEscape` value escaping. It deliberately does *not* parse TOML — it operates on lines, with the real parser (`LoadManifestContext`) having validated the file moments earlier.
- **`LWPT.Install` keeps the lockfile-diff pruner exported** (`PruneOrphanedPackages`) for its unit tests, but the only production caller is the mutation transaction itself — archive naming (`<name>-<safe-ref>.tar.gz`) is the installer's private knowledge and must not be duplicated.
- **A dep added with include/exclude globs still requires hand-editing** (the inline-table form is the advanced path per ADR-0009, unchanged). `add` covers the bare-string canonical form only.
- The decision **forecloses** nothing structural — both subcommands are frontends over the transaction; retiring them would strand no on-disk format. It **opens** a future "prune on every install" follow-up ADR and a `lwpt add --dev`-style dependency-class split if one ever materializes (none planned; LWPT has a single dependency class today).
