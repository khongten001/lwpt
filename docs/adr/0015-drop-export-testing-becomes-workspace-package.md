# Drop `lwpt export`; testing framework graduates to `packages/testing/` workspace package

`lwpt export <unit>` was a single-target subcommand that extruded an embedded copy of `TestingPascalLibrary.pas` + `Shared.inc` into the consumer's `.lwpt/modules/` directory. earlier it earned its keep because the testing library was the only thing every consumer needed but could not be a regular dep — LWPT had no package system that worked for self-hosted use, so the framework rode along inside the LWPT binary as a generated `LWPT.Embedded.TestingLibrary.inc` blob (regenerated by `scripts/embed.pas` via the `[prebuild]`/`[pretest]` hook in the root manifest). `CmdTest` even auto-extruded on first run so the verb itself was rarely typed. the workspace-package machinery (ADR-0014 amendment "Workspaces") made the whole arrangement redundant: a package consumed via `[workspaces] include = ["packages/*"]` is symlinked into `.lwpt/modules/<name>/` exactly the same way the extrude path used to write `.lwpt/modules/TestingPascalLibrary.pas` — minus the binary-embedded blob, the staleness hook, the auto-extrude branch in `CmdTest`, and the `lwpt export` subcommand. This change removes the lot: the testing framework graduates into `packages/testing/` as the fifth workspace package alongside `httpclient`, `cli`, `semver`, `toml`; `CmdExport` + `HandleExport` + `ExtrudeTestingLibrary` + `ExtrudeUnit` + the embedded `.inc` + `scripts/embed.pas` are deleted; the subcommand surface shrinks from eight to seven (the Hard Constraint in `AGENTS.md` updates accordingly). The existing framework canary (`source/Tests.TestingPascalLibrary.Canary.Test.pas`) moves with the package and is renamed to the conventional `packages/testing/source/TestingPascalLibrary.Test.pas`, matching every other package's `<name>.Test.pas` self-test. Hard removal, no deprecation shim — `lwpt` is still pre-1.0 and the migration for any external consumer is one line in `lwpt.toml`.

## Considered Options

### Removal vs. retention

- **Keep `lwpt export` as-is** — the verb continues to exist; the embedded blob continues to ride along in the binary. Rejected: every value proposition of `export` is now delivered by the workspace-package + zero-install combo, and keeping a no-payoff subcommand bloats the surface (8 → 7 is real), forces continued maintenance of `scripts/embed.pas` + the `.inc` regen hook, and complicates the mental model ("everything is a package, except the testing library which is also magically embedded").
- **Keep `lwpt export` with a deprecation shim** — for one release, `lwpt export testing` warns + auto-runs the install path, then deletes in the next. Rejected: the half-life of confusion exceeds the benefit. Pre-1.0 projects can break callers; one-line migration in `lwpt.toml` is cheaper than maintaining a stub.
- **Hard removal.** *Chosen.* Delete the subcommand, the handler, the embedded blob, the regen script, the auto-extrude path, and the staleness hook in the root manifest. The five-package monorepo (`httpclient`, `cli`, `semver`, `toml`, `testing`) is the new steady state.

### Naming + location

- **`packages/testing/`** as the fifth workspace package, matching the four existing ones (`httpclient`, `cli`, `semver`, `toml`). *Chosen.* Consistent with ADR-0014's pluralised noun naming + auto-discovered-via-glob layout.
- **`packages/testinglibrary/`** preserving the verbose vendored name. Rejected: ADR-0014 already established short package names independent of the file names inside (`packages/cli/` contains `CLI.Help.pas` + `CLI.Options.pas` + ..; `packages/toml/` contains `TOML.pas` + `OrderedStringMap.pas` + ..). `testing` follows the same pattern.
- **`packages/tpl/`** as a three-letter acronym. Rejected: `testing` is what consumers think of the framework as; `tpl` reads as Template, not Testing Pascal Library.

### Canary test location + naming

- **Move the canary to `packages/testing/source/TestingPascalLibrary.Test.pas`.** *Chosen.* Self-tests live with the unit they test (Q16 from ADR-0014). The PascalCase `<UnitName>.Test.pas` form matches `HTTPClient.Test.pas` + `Semver.Test.pas`.
- **Keep the canary in `source/Tests.TestingPascalLibrary.Canary.Test.pas`** as a LWPT-wide tier-0 sanity check. Rejected: the canary tests the testing framework's own invariants — it belongs *with* the framework, not in the consumer's own source. earlier its placement in `source/` was an artefact of the framework being embedded.
- **Rename to `Canary.Test.pas`** dropping the unit-name prefix. Rejected: less greppable + breaks the `<Unit>.Test.pas` convention every other package follows.

### What replaces the auto-extrude UX

earlier, dropping a `Foo.Test.pas` into a brand-new project and running `lwpt test` Just Worked because `CmdTest` auto-extruded the testing library on first call. With `export` gone, the consumer needs `testing` in `[dependencies]` (or `[workspaces]` for monorepo cases) + a `lwpt install` run.

- **`lwpt init` carries the ergonomics.** *Chosen.* ADR-0010's interactive init flow already prompts for project metadata; Adds a `Set up testing library? [Y/n]` prompt that adds `testing = "<source>"` (TBD: workspace-protocol form or git-host form, depending on whether the consumer is a monorepo or a standalone project) to the scaffolded manifest + scaffolds an example `<entry>.Test.pas`. With `--yes`, default yes. New projects go from `lwpt init --yes` straight to `lwpt test` working with zero extra steps.
- **A warning in `lwpt test`'s "no testing library found" error** pointing at the dep + the init flow.
- *Not chosen:* automatic dep injection. Adding `testing` to the manifest silently from `lwpt test` would surprise the user; explicit dep declaration is the right contract.

### Self-consumption inside LWPT's own monorepo

- **`packages/testing/` as a workspace package, auto-discovered.** *Chosen.* `[workspaces] include = ["packages/*"]` already in the root manifest picks it up; `lwpt install` symlinks it into `.lwpt/modules/testing/`; the cfg emitter wires `-Fu .lwpt/modules/testing/source/ -Fi .lwpt/modules/testing/source/`. Same shape as the four existing packages, zero new mechanism.
- **`source/TestingPascalLibrary.pas` stays in `source/`** as a tier-0 unit alongside `LWPT.Core.pas`. Rejected: it's not LWPT-owned; it's vendored from GocciaScript. Keeping it in `source/` post-graduation would mean the LWPT-internal vendored unit (`source/TestingPascalLibrary.pas`) and the external workspace-package vendored unit (`packages/testing/source/TestingPascalLibrary.pas`) diverge over time. One canonical home.

### Consumer migration (external projects, post-amendment)

- **No automated migration tool.** *Chosen.* `lwpt export testing` consumers add one line to their `[dependencies]` (or `[workspaces]`) + run `lwpt install`; nothing else changes (their `*.Test.pas` files keep using `uses TestingPascalLibrary;` unchanged because the library's unit name didn't change). One-line migration doesn't earn a migration tool.
- *Not chosen:* a `lwpt repair --migrate-from-export` flag that auto-rewrites the manifest. The mechanism would have to inspect existing `.lwpt/modules/TestingPascalLibrary.pas` (the earlier extruded artefact) + write a dep entry. Too much code for a one-line edit.

### Hook denylist + lifecycle-hook ADR

ADR-0011's denylist (the explicit set of subcommands that may NOT have hooks) included `export`. With `export` gone the denylist entry is stale; ADR-0011 gets amended to remove the `export` row + the supporting paragraph ("a one-shot 'give me the embedded testing library' operation"). The five remaining denylisted subcommands stay denylisted: `format`, `repair`, `init`, `run`, plus the implicit "anything not in the allowlist".

## Consequences

- **One fewer subcommand.** `AGENTS.md`'s Hard Constraint on subcommand surface updates from 8 to 7. The Quick Reference table loses the `Extrude the testing library` row. README.md's subcommand block + quick-start example drop the `lwpt export testing` step.
- **Smaller `lwpt` binary.** The embedded `LWPT.Embedded.TestingLibrary.inc` was ~16 KB of generated Pascal-string literal content. Trimmed.
- **Simpler bootstrap.** `scripts/bootstrap.pas` loses the `EmbedIsStale` / `RegenerateEmbed` logic; the script becomes single-purpose (compile lwpt). The shell + batch wrappers add `-Fupackages/testing/source -Fipackages/testing/source` to mirror the four existing package paths.
- **Root manifest loses its [prebuild]/[pretest] hooks.** The only entries were the staleness-gated `scripts/embed.pas` regen; with no embedded blob, nothing to regenerate. The sections delete entirely — the manifest's hooks story is now "available for projects that need it, unused here".
- **Self-host story is uniform.** earlier the testing library was the one exception to "everything is a package". Post-amendment the rule has no exceptions.
- **External `lwpt export testing` users break.** They get a `unknown subcommand: export` error on first invocation. Mitigation: the release notes call out the one-line migration explicitly; the CHANGELOG entry is one of the bullets surfaced in the GitHub Release.
- **`docs/vendored.md` table updates.** `TestingPascalLibrary.pas` moves from "Stays in `source/`" / "Phase 3 candidate" to the `packages/testing/` row + "Phase 1 complete; Phase 2: graduate `testing` as a standalone repo". Phase 3 of the graduation roadmap loses one entry (now: `Platform` only).
- **ADR-0011's denylist paragraph trims.** `export` row + the supporting prose disappear. The remaining denylist (format/repair/init/run) keeps its rationale.
- **ADR-0014's "Packages list" amends.** Adds the `packages/testing/` entry to the in-repo extraction table.
- **CI workflows add one `-Fu`/`-Fi` pair.** `ci.yml` + `release.yml`'s cross-build step adds `-Fupackages/testing/source -Fipackages/testing/source`. No new toolchain work; cache-version stays at v1.
- **The framework canary moves with the package + is renamed.** `source/Tests.TestingPascalLibrary.Canary.Test.pas` → `packages/testing/source/TestingPascalLibrary.Test.pas`. The test body is unchanged; the doc-block updates the framing (it's the package's self-test now, not a tier-0 framework canary embedded in LWPT's own `source/`).
- **`.lwpt/modules/`'s earlier extruded artefacts (`TestingPascalLibrary.pas` + `Shared.inc` as loose files) are deleted.** The fifth symlink `.lwpt/modules/testing -> packages/testing/` is created by `lwpt install` on the post-amendment run; users with an old extruded copy need to re-run install once.

## Migration (existing external consumers)

If you depended on `lwpt export testing` to vendor the testing library into your project:

1. Add `testing` to your manifest's `[dependencies]`:

   ```toml
   [dependencies]
   testing = "frostney/lwpt-testing@^1.0.0"   # Phase 2 form, post-graduation
   # or, until Phase 2 lands:
   testing = { source = "frostney/lwpt@^0.1.0", include = ["packages/testing/**"] }
   ```

2. Run `lwpt install`.
3. Delete the earlier extruded `.lwpt/modules/TestingPascalLibrary.pas` + `.lwpt/modules/Shared.inc` (the new symlink at `.lwpt/modules/testing/` supersedes them).
4. Your `*.Test.pas` files need no changes — `uses TestingPascalLibrary;` resolves through the new path via cfg.

Phase 2 graduation of `packages/testing/` to its own `frostney/lwpt-testing` repo follows ADR-0003's roadmap; until then the `include = ["packages/testing/**"]` escape hatch on the main LWPT repo is the documented form.
