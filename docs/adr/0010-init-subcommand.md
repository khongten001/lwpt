# `lwpt init` is the seventh subcommand

The subcommand surface, frozen at six since an earlier cycle ([Hard Constraint in AGENTS.md](../../AGENTS.md)), expands to seven with `lwpt init`. The new subcommand scaffolds a fresh LWPT project in the current directory: it writes a minimal `lwpt.toml` (interactive prompts for **name**, **version**, **source folder**, **build folder**, and **project entry name**), a matching `source/<entry>.pas` hello-world program, and `.gitignore` entries for `.lwpt/tmp/`, `.lwpt/install.lock`, and the configured `<build-dir>/`. After scaffolding, the interactive flow prompts (Y default) to run `lwpt install` + `lwpt build` so the user lands on a working executable they can run on the first try. With `--yes` (or the short `-y` alias) it skips all prompts, uses sensible defaults derived from the current directory name, and **does not** auto-run the install/build chain — the user opted into a scripted setup, so they drive the rest. `--force` overwrites an existing `lwpt.toml`; without it, an existing manifest is a hard error so accidental re-init can't destroy a project. We chose this over keeping the surface at six and documenting `cp ./templates/lwpt.toml.example .` because the npm/cargo precedent of one-command project bootstrap is too valuable to skip — `lwpt init` from an empty dir → answer five prompts → "hello from \<name\>" is the cleanest first-run story we can offer, and every LWPT user does it once on each project.

## Considered Options

- **Keep the surface at six; ship an example manifest template instead.** Smallest change; no new code. Rejected because `lwpt init` is the canonical entry point in every npm/cargo/yarn workflow — users land on the LWPT docs already mentally indexed against that command, and "you have to copy a file" is a documentation friction not a usability win. The Hard Constraint about subcommands being frozen exists to prevent random additions; `init` is the one missing piece every comparable tool has.
- **`lwpt init` non-interactive only (always `--yes` semantics).** Single mode, less code. Rejected because new-project bootstrap is exactly the spot where prompts pay off — getting the name / version / source / build / entry name right at creation time saves a manual edit pass. The cost (a few hundred lines for the prompt loop) is bounded.
- **`lwpt init` is interactive only; no `--yes`.** Forces every user through the prompts. Rejected because CI / scripted setups need a non-interactive escape hatch (`npm init -y` is the universal precedent).
- **Scaffold the manifest only; no source file, no [targets] entry.** Less magic; LWPT stays maximally unopinionated. Rejected because the resulting project doesn't actually do anything — the user has to invent a source file, wire up `[targets]`, and figure out the `program X;` / unit-vs-program distinction before `lwpt build` does something visible. The shipped scaffold is intentionally tiny (one `program <name>;` block, one `WriteLn`), but it produces a runnable binary on `lwpt build` and that's the win.
- **Write `lwpt.lock` as part of init.** Symmetric with the manifest. Rejected because `lwpt install` is the authority on lockfile shape (schema version, resolved refs, archive hashes — all of which `init` would have to stub). An empty stub buys nothing and risks divergence the day the schema rev moves; letting `install` create the file on first run keeps a single owner.
- **Run `lwpt install` + `lwpt build` unconditionally after init (no prompt).** Maximally hands-off. Rejected because the first-time user may want to inspect the scaffold before triggering network fetches (even though the default scaffold has zero deps, the user might add them between `init` and the first `install`). A Y-default prompt is the right balance — one keystroke for the common case, easy to skip for the inspect-then-install path.

## Consequences

- **The "subcommand surface is frozen" Hard Constraint gains an ADR-cited exception.** AGENTS.md updates to mention init in the canonical seven; the constraint still bars random additions (each new subcommand needs its own ADR).
- **`lwpt init` refuses to clobber.** An existing `lwpt.toml` in CWD is a hard error unless `--force` is passed. The error message names the existing file so the user can `rm` if they meant to recreate.
- **`.gitignore` is created if absent, or appended-to if present.** We only add lines that aren't already in the file (idempotent — running `init` twice doesn't duplicate entries). Three entries are written by default: `.lwpt/tmp/`, `.lwpt/install.lock`, and the configured `<build-dir>/`.
- **A working `[targets]` entry is auto-created** that points at the scaffolded `source/<entry>.pas`. The first `lwpt build` produces `<build-dir>/<entry>` and the user can run it to see "hello from \<name\>" — the smallest possible feedback loop.
- **The `program <name>;` declaration is sanitised for Pascal identifier rules** (hyphens → underscores). Package names happily contain hyphens (npm convention; LWPT inherits), but Pascal identifiers don't, so a project named `my-app` gets `program my_app;` in `source/my-app.pas`. FPC doesn't require the program name to match the filename, so this stays invisible to the user. The greeting string + target name keep the original spelling.
- **`lwpt init --yes` is the scripted-setup path** and intentionally stops at the scaffold; it does *not* auto-run install + build. Interactive mode prompts (Y default) before running them, so the common interactive flow is "init → answer prompts → confirm install+build → working binary".
- **No `lwpt.lock` is written by `init`.** The file is created by the first `lwpt install` (whether triggered by init's post-action prompt or by the user later). This keeps lockfile shape — schema version, package entries, hashes — owned by a single command.
- The decision **forecloses** the six-subcommand surface; reverting is theoretically possible but would break every `init`-bootstrapped project. The decision **opens** future "lwpt init --template \<name\>" extensions if real fixture-style templates emerge (we'd ship them alongside the binary). For v1 the only templates are the default scaffold (interactive prompts + hello world + install/build chain) and the `--yes` skip path.
