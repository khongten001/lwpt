# Lockfile schema v2 — archiveHash is a sibling of computedHash

The `lwpt.lock` file records, per resolved dep, both the SHA-256 of the on-disk extracted modules tree (`computedHash`) **and** the SHA-256 of the source `.tar.gz` archive in the cache (`archiveHash`). Both fields are required in v2; `archiveHash` is the empty string for `skLocal` deps that have no archive. Earlier schema (v1) recorded only `computedHash`; v1 lockfiles are rejected at load time with a clear migration hint ("delete and re-run `lwpt install`"). The schema number is stored as `version = 2` at the top of the file. We chose this over keeping a single `computedHash` field because `--frozen` mode is supposed to be a tamper-detection gate (per ADR-0002), and tamper detection that ignores the archive bytes lets an attacker swap the cached tarball under a legitimate-looking modules tree.

## Considered Options

- **Keep schema v1 (only `computedHash`)**. Simpler, no migration. Rejected because `--frozen` then can't tell whether the archive in `.lwpt/archives/` matches what produced the current modules tree — the two could disagree if the archive was tampered with after install. Both halves are needed for a "this is exactly what we resolved" guarantee.
- **Single combined hash (concatenate archive bytes + tree contents, then SHA-256)**. One field, one comparison. Rejected because it loses the ability to diagnose *where* tampering happened. With two fields, a frozen-mismatch report can say "the archive matches but the tree was edited" vs "the archive itself was swapped," which is the difference between a developer accident and a supply-chain attack.
- **`archiveHash` optional, default-skip when absent**. Backward-compatible with v1 lockfiles; no migration error. Rejected because silent skipping defeats the security purpose — a v1 lockfile would be auto-trusted on a fresh checkout. Explicit migration is the safer default.
- **Bump to `version = 2` AND introduce a `[meta]` table for future-proofing**. Cleaner schema, but adds parsing surface (the partial TOML reader's table-of-inline-tables limitation means table evolution is tricky) for no v1 benefit. Defer until v3 needs it.

## Consequences

- **v1 → v2 is a one-shot migration**: delete `lwpt.lock`, run `lwpt install`. The error message says exactly that. No auto-migration code path — the next `install` regenerates from `lwpt.toml` deterministically anyway.
- **`--frozen` is now a two-hash gate**: archive bytes AND extracted tree, both must match. A user who manually edits files under `.lwpt/modules/` (e.g., debugging a vendored dep) and then runs `--frozen` gets `EVerifyError` naming the tampered dep. The recovery is `lwpt install` (non-frozen) to regenerate the tree from the archive — or, if the archive itself was edited, restore the archive from version control.
- **Local-source deps have empty `archiveHash`**. The verifier skips archive comparison when either side is empty; mismatch-on-one-side still raises. This is a soft spot: an attacker who swaps a local-source dep's tree on disk WITH a matching computedHash update to the lockfile evades detection. The full fix is signing the lockfile itself, which is out of scope for v1.
- **The `[generated]` machinery is unaffected.** It re-hashes inputs and re-runs the generator script; the lockfile schema change is orthogonal.
- The decision **forecloses** v1-shape lockfiles for any future LWPT version; reverting would require treating older lockfiles as silently broken, which the migration-error path explicitly rejects. The decision **opens** the door to per-dep extra metadata (signing, source-of-trust, fetched-at timestamp) — all future-additions can go alongside `archiveHash` without another schema bump, as long as readers tolerate unknown keys (which our TOML reader does by default).
